# 治法与学术证据

[[context-failure-modes]] 把"context 出事"解剖成 4 种崩法 — Poisoning（中毒）/ Distraction（分心）/ Confusion（混淆）/ Clash（冲突）。但只知道病因不解决问题。这一篇拆 Drew Breunig<Cite id="breunig2025fixes" /> 总结的 **6 种治法**，挨个对症下药 — 并给出每一种治法的跨论文实证（Chroma 18 模型测试、Anthropic 多 agent、工具数量崩坏曲线、Provence pruner 等）。

读完这一篇，你 debug 一个 agent 的 context 问题时，能给出 "它是哪种崩法 → 用哪种治法 → 这种治法在 X paper 上的实证效果" 这种结构化的诊断。

## 1. 六种治法的总览

先把 6 种治法和它们对应的崩法摆一张表：

| 治法 | 一句话 | 主治哪种崩法 | 不适合 |
|---|---|---|---|
| **RAG** | 只把相关信息检索进 context | Distraction / Confusion | 信息不够结构化、retriever 质量差 |
| **Tool Loadout** | 只加载这次相关的工具定义 | Confusion | 任务真的需要很多工具 |
| **Context Quarantine** | 把 context 拆进独立 agent 各自的窗口 | **所有 4 种** | 子任务强耦合 |
| **Context Pruning** | 删 context 里无关的部分 | Distraction / Clash | 关键信息和无关信息难区分 |
| **Context Summarization** | 把累积的 context 浓缩 | Distraction | **Poisoning（会把毒一起腌进去）** |
| **Context Offloading** | 把信息存到 context 外（笔记 / scratchpad） | 全部 | 不能离线访问外部存储的场景 |

注意 **Summarization 对 Poisoning 是 "disease-cure mismatch"** — 这条 [[context-failure-modes]] 里讲过，**用错药比不用药更糟**。下面挨个展开 6 种治法的细节 + 实证。

## 2. RAG — 检索回相关的，剩下别塞

RAG 在 [[memory-basics]] 里讲过机制（embedding + 向量库 + ANN 检索）。这里讲它作为 **context 治法**的角度：

**问题**：你有 100 个产品文档，但用户一次只问 1 个产品。

**坏做法**：把所有 100 个文档都塞进 system prompt（"以下是产品信息..."）。

**RAG 做法**：

1. 把 100 个文档全部 embedding 进向量库（一次性预处理）
2. 用户问"产品 X 怎么用" → embedding 用户问题
3. 在向量库里找 top 3 最相关的文档片段
4. 只把这 3 段塞进 context

效果：context 从 100 个文档（几十万 token）→ 3 段（几千 token）。

### Chroma 18 模型实证：RAG 不只是省 token，是必须

Chroma Research<Cite id="chroma2024rot" /> 在 18 个前沿模型上做了一个直接对比实验：

| 方式 | 给的 context | 准确率 |
|---|---|---|
| **Full input** | 全部相关上下文一起塞 | 较低 |
| **Focused-only** | 只塞和当前问题最相关的子集 | **显著更高** |

这个结果在 18 个模型上一致 — 包括 Claude / GPT / Gemini 等顶级模型。意味着 RAG 不只是"省 token"工程优化，**全塞反而让模型表现更差**。这是 [[context-failure-modes]] 里 Distraction（分心）在生产实证上的根。

## 3. Tool Loadout — 只加载这次能用上的工具

很多人对 agent 工具的本能反应是 "**多给点工具，让 agent 啥都能干**"。这是个典型陷阱。

### 一个标志性数字：工具越多，agent 越差

| 模型 | 多少工具时开始崩 |
|---|---|
| **DeepSeek-v3** | 超过 30 个工具开始混淆，超过 100 几乎必败 |
| **Llama 3.1 8B** | 用 46 个工具会崩；只用 19 个就能正常工作 |
| **GPT-4 / Claude** | 不同任务上从 20 个工具开始下降 |

注意 Llama 3.1 8B 的数据 — **46 工具崩 vs 19 工具正常**。这不是渐变下降，是**断崖**。

### 工具数量崩坏的根因

不是模型笨，是**工具描述全部塞进 system prompt 后挤掉了实际任务的 context 空间**。一个典型工具描述 200-500 token，46 个工具就是 10K-20K 进 prompt — 还没开始干活，attention budget 已经被消耗一大半。

### Tool Loadout 的做法：动态选

不要把所有工具的定义都丢给 agent。**按任务动态选这次该加载哪些**：

```text title="Tool Loadout 模式"
用户问: "查一下 Q3 销售数据"

Tool Loader (一个 LLM 或规则):
  → 这个问题应该用 数据查询 / 报表生成 类工具
  → 加载: search_db, generate_report, format_table
  → 不加载: send_email, schedule_meeting, edit_file, ... (其他 30 个)

Agent 看到的 tools 参数: 只有 3 个工具,不是 33 个
```

### 实证：动态选工具能 +44%

Anthropic 自己跑过的对比 — 在一个 agent 系统中加入 **dynamic tool loadout**（按任务动态选工具）vs 全工具集，**任务成功率 +44%**。

这个 +44% 是免费午餐 — **不动模型、不改 prompt、只改"该展示哪些工具"**。

## 4. Context Quarantine — 隔离

这是 [[long-running-techniques]] 里 **sub-agent** 那条治法 — 把 context 拆进独立 agent 各自的窗口。

**为什么它能治"所有 4 种崩法"**：

- 治 **Poisoning** — 中毒只在 quarantine 区域内传染，不会进主 agent
- 治 **Distraction** — 每个子 agent 只看自己负责的窄 context
- 治 **Confusion** — 每个子 agent 只需要它任务相关的工具子集
- 治 **Clash** — 矛盾信息被关在不同子 agent 的窗口里，主 agent 只看蒸馏摘要

### 实证：Anthropic 多 agent +90.2%

Anthropic 在 *How we built our multi-agent research system*<Cite id="anthropic2024multiagent" /> 里报告 — 在复杂研究任务上，**lead agent + worker sub-agents 架构**（每个 worker 用干净的窗口处理一个子查询）比单 agent 高 **90.2%**。

这个数字大到让人想"那干脆全用多 agent"。但 [[multi-agent-debate]] 那一篇会告诉你为什么不能这么草率 — 多 agent 引入它自己的失败模式 **context fragmentation（碎片化）**，子 agent 之间信息对不齐、决策互相打架。

## 5. Context Pruning — 删无关的

如果你不能或不想用 quarantine（任务耦合度高），可以用**剪枝** — 直接把 context 里无关的部分**删掉**。

### 用一个专门的 LLM 做 pruning

最简单的实现：在每轮 LLM 调用之前，让一个**小的、快的 LLM** 读 context 并删掉无关部分。

```text title="Pruner 流程"
Round N 的 context (50K token):
  - system_prompt (1K)
  - 5 轮历史对话 (20K)
  - 检索回来的 10 段文档 (25K)  ← 但只有 2 段真的相关
  - 用户当前问题 (200 token)
        ↓
Pruner LLM (轻量, 比如 Haiku):
  "看完整个 context, 标出哪些段对回答当前问题没用"
        ↓
删除标记的段, 留下:
  - system_prompt (1K)
  - 最近 2 轮历史 (8K)
  - 真正相关的 2 段文档 (5K)
  - 用户问题 (200 token)
        ↓ 14K vs 50K, 节省 72%
Main LLM 接着推理
```

### 实证：Provence pruner

Provence<Cite id="laskar2024prune" /> 是一个专门为 RAG 设计的 pruner — 它能砍掉一篇文档 **95% 的内容**还保住关键信息。

这个数字震撼之处在于 — 它意味着 **绝大多数被检索回来的内容是噪音**。真正有用的信息可能只占检索结果的 5%。

### 什么时候适合用 pruning

- 检索器质量一般（召回率优先于精度）
- 任务对**回答精确度**敏感（不能让无关内容干扰）
- 你愿意为 pruner 多付一个 LLM 调用的成本

## 6. Context Summarization — 压缩

[[long-running-techniques]] 里讲过 compaction — 这是它在 context 治法语境下的同款思路：**把长历史摘要成短摘要**。

但**记住 [[context-failure-modes]] 里那个反转** —

> [!warning] Summarization 治 Distraction，但**不治 Poisoning**
> 摘要的本质是"忠实地浓缩 context"。如果 context 里有一条幻觉/错误结论（Poisoning），摘要器会**忠实地把这条错误也浓缩进去**继续当真理引用。
>
> 对中毒的 context，唯一可靠的解药是**开一个全新的 fresh session 把毒源彻底丢掉**（这正是 quarantine 的另一种形态）。
>
> 用错药的代价不是没效果，是**把毒固定下来**。这是 "disease-cure mismatch" 的最经典案例。

## 7. Context Offloading — 卸载到 context 外

最后一种治法：**把信息从 context 里搬出去，存到外部**（笔记 / scratchpad / 数据库），需要时再拉回来。

这就是 [[long-running-techniques]] 里讲的 **note-taking** 在治法语境下的同款思路。

### 一个具体收益：Anthropic 的 "think" 工具

Anthropic 在他们的 coding agent 里加了一个简单到反直觉的工具 — **`think(thought)`**：让 agent 把"中间思考"显式写出来作为一个 tool call，而不是放在主推理流里。

效果：在 SWE-bench 任务上 **+54%**。

为什么这么好用？

- agent 的中间 thought 不再挤占主 context 的注意力预算
- 这些 thought 被记录下来 — 后面可以引用、可以分析、可以被 compaction 安全压缩
- agent 长出"显式思考"的习惯 — 比"在 prompt 里悄悄推理"更可控

这正是 [[reflexion]] 里 self-reflection 的轻量版本 — **让 agent 把内部状态外化**就能解决一类问题。

## 8. 一个 debug 决策树

最后给一个**实用 debug 流程** — 当你的 agent 出问题时，按这个流程定位治法：

```text title="Agent context 问题诊断流程"
Agent 表现差 →
  ↓
Q1: 它是不是在重复用已经被证明错的信息?
  YES → Poisoning. 用 Quarantine (开新 session 或 sub-agent)
        不要用 Summarization (会腌进去)
  NO  → 继续
  ↓
Q2: 它是不是 "在历史里翻找" 而不是综合推理?
  YES → Distraction. 用 RAG (只塞相关) + Summarization + Pruning
  NO  → 继续
  ↓
Q3: 它是不是选错工具 / 工具描述混乱?
  YES → Confusion. 用 Tool Loadout (动态选工具) + 命名空间
  NO  → 继续
  ↓
Q4: context 里是不是有互相矛盾的指令?
  YES → Clash. 用 Quarantine + 集中放 instruction (不要分散)
  NO  → 继续
  ↓
Q5: 是不是 token 总量超过窗口?
  YES → Offloading (写笔记) + Summarization + Compaction
  NO  → 可能不是 context 问题. 检查模型本身能力 / 工具质量 / prompt
```

这张表是 [[context-failure-modes]] 4 种崩法和本篇 6 种治法的**直接组合 debug 表**。
