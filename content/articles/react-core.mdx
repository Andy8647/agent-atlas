# ReAct: 推理与行动交替

接 [[workflow-vs-agent]]。前面我们把 agent 主循环抽象写出来了：

```python title="agent 的主循环（抽象版）"
history = [system_prompt, user_task]
while not done:
    step = llm(history)        # ← 这一行
    ...
```

整本讲义后面所有章节，本质上都是在拆解**这一行**。模型看着历史，决定下一步做什么 — 它具体怎么决定？拍脑袋？掷骰子？这是一个完全没展开的黑盒。

ReAct<Cite id="yao2023react" /> 给的是单 Agent 范式里**最朴素**的答案。但你看到答案之前得先看明白：**这个问题的两种最直觉的解法分别会怎么死**。先把死法看清楚，ReAct 那个"朴素到像废话"的招你才会真正服气。

> [!note] 为什么从失败讲起
> 这套讲义后面整整十节都是在治各种失败模式 — Plan-and-Solve 治"没全局规划"、Reflexion 治"重复循环"、Context Engineering 治"长上下文崩坏"、Harness 治"工具流水线出错"。**记住失败比记住成功更重要**：你只有先看清楚哪里会崩，才知道每个新技术到底在补哪个洞。

## 1. 两种朴素做法，各自怎么崩

把 `step = llm(history)` 这行直接实现，朴素的选择只有两个：要么让模型**只动**（每步只吐一个 action），要么让模型**只想**（一口气想到底再给答案）。这两种各自都有响当当的名字 — 一个叫 Act-only，一个叫 Chain-of-Thought（CoT）。它们也都会以一种很对称的方式死掉。

### 死法 A：纯行动（Act-only）— 不长记性的瞎试

最直接的想法：让模型每一步直接吐一个 action，不准它废话。论文里在 ALFWorld<Cite id="shridhar2020alfworld" />（文字版"在房间里找东西放进抽屉"的小游戏）记录的轨迹是这样的：

```text title="ALFWorld · Act-only 轨迹"
任务：把 pepper shaker（胡椒瓶）放进抽屉

Act 1: Go to drawer 1            → Obs: The drawer is closed.
Act 2: Open drawer 1             → Obs: 抽屉空的，只有 dishsponge 和勺子
Act 3: Go to sinkbasin 1         → Obs: 水槽边有 dishsponge、spatula（没有 pepper shaker）
Act 4: Take peppershaker from sinkbasin → Obs: Nothing happens.
Act 5: Take peppershaker from sinkbasin → Obs: Nothing happens.  ← 又试了一遍 ✗
```

看 Act 4 → Act 5。环境已经明明白白告诉它"Nothing happens"了，水槽里就是没有胡椒瓶。但模型下一步**又做了同一个动作**。它不是没看到反馈 — 反馈就拼在 history 里；它是没有一个步骤把那个反馈**消化成"水槽里没有，我得换地方找"这种判断**。

> [!note] 这个死法告诉你什么
> 纯行动这条路缺的不是"看见"，是"想"。环境反馈进来了，但没有任何一步把它转化成对状态的理解和对下一步的判断。模型就退化成了一个**没记性的、机械重试的执行器**。

### 死法 B：纯推理（Chain-of-Thought）— 自信地编

另一个极端。让模型一口气 "think step by step" 想到底，中途**不接触外部世界**。这就是你熟悉的 CoT<Cite id="wei2022cot" />。论文 Figure 1 的 HotpotQA<Cite id="yang2018hotpotqa" /> 例子：

```text title="HotpotQA · Chain-of-Thought 轨迹"
问题：除了 Apple Remote，还有什么设备能控制 Apple Remote 最初设计要交互的那个程序？

Thought: Apple Remote 最初是设计来控制 Apple TV 的。
         Apple TV 可以被 iPhone、iPad、iPod Touch 控制。
         所以答案是 iPhone、iPad、iPod Touch。
Answer: iPhone, iPad, iPod Touch   ✗
        （错了 — 那个程序是 Front Row，不是 Apple TV）
```

错就错在 Thought 第一句。Apple Remote 当年其实是设计来控制 Mac 上一个叫 **Front Row** 的媒体中心程序，不是 Apple TV。模型把这个事实记岔了，但**没有任何机会去核实**，于是后面整条推理链全部基于这个错误一路滚下去，给出一个一本正经的错答案。

论文里管这个叫 **static black box**（静态黑箱）：

> The model uses its own internal representations to generate thoughts and is not grounded in the external world, which limits its ability to reason reactively or update its knowledge. This can lead to issues like **fact hallucination** and **error propagation**.
>
> 模型只靠内部表示生成想法，不与外部世界 grounded — 既不能反应式地推理，也不能更新自己的知识。结果就是 **事实幻觉** 和 **错误传播**。

> [!important] 两种死法是镜像对称的
> | 范式 | 会的事 | 缺的事 | 典型死法 |
> |------|--------|--------|----------|
> | 纯行动（Act-only） | 接触现实 | **不会想** | 观察消化不了 → 瞎试、重复 |
> | 纯推理（CoT） | 会想 | **不接触现实** | 一步错、步步错 → 自信幻觉 |
>
> 一个缺"想"，一个缺"接触现实"。ReAct 全部的聪明，就是让这两样**在每一步互相纠偏**。

## 2. ReAct 的核心招：把"语言"也算成一种动作

ReAct 论文的形式化定义只有一行，但这一行就是整篇论文的灵魂。先看原文：

> **Definition (ReAct's augmented action space)**：把动作空间从"真实动作 $A$"扩展成 $\hat{A} = A \cup L$，其中 $L$ 是**语言空间**。语言空间里的一个动作 $\hat{a} \in L$ 叫做 **thought（想法）**。它不改变外部环境（因此没有 observation 反馈），只基于当前 context 做推理，**更新 context** 以支持后续的推理或行动。

这个定义乍看挺玄。翻译成工程直觉就一句话：

> **Thought 就是一个"只往 history 里追加文字、不调任何 API、没有副作用"的内部动作。**

它不删文件、不发 HTTP 请求、不查数据库 — 它做的事情只有一件：**往 history 里追加一段"我现在怎么想"**。但**正因为追加进了 history**，下一次循环的 `llm(history)` 就能看着这段想法、做出一个比上一次更好的决定。

这是 ReAct 最反直觉但最关键的一步：**它没有发明任何新机制**。模型本来就在做 `llm(history)`，本来就会输出文字 — ReAct 只是给了一类输出（thought）一个明确的语义位置：**它是一个不接触世界、但会影响下一次决策的内部动作**。

于是 L0 那个抽象的 loop，被填成了具体的 **Thought → Action → Observation** 交替：

```python title="agent-loop.py · ReAct 版本" {3-6}
history = [system_prompt, user_task]
while not done:
    thought = llm(history)        # 语言动作：无副作用，只更新 context
    action  = llm(history)        # 真实动作：调工具
    obs     = execute(action)     # 环境反馈
    history += [thought, action, obs]
```

注意第 3 行和第 4 行其实是同一个 `llm(history)` 调用 — 现代实现里模型一次就把 thought 和 tool_call 一起吐出来。但概念上把它拆成两步更清楚：**模型先在脑子里走一遍，再决定动手做什么**。

### 看一条正确轨迹

回到 Apple Remote 那道题。这次让 ReAct 跑一遍，每一步都可以看到 Thought / Action / Observation 三栏 — 用下面这个组件点 "下一步" 看：

<ReActVisualizer trajectory="apple-remote" />

你重点看 **Thought 2** 和 **Thought 3**：

- 第一次搜索后，模型在 Thought 2 里发现"原来是 Front Row，不是 Apple TV"，**主动调整了下一步的搜索方向**
- 第二次搜索"Front Row"没找到，Thought 3 里模型**自己重新组织查询**："那就搜 Front Row (software)"

这两个动作纯行动做不到 — 它没有"想"那一步，遇到失败的反馈只会瞎试。纯推理也做不到 — 它根本没机会接触搜索引擎的真实结果，只能基于第一次的错误印象一路编下去。**ReAct 就在这种交替里把两种死法的洞同时补上**。

## 3. 为什么"缝起来"就管用 — 双向协同

论文标题里那个 **Synergizing**（协同）是字面意思 — 两个方向**互相喂**：

> [!tip] Reason ↔ Act 双向协同
> - **Reason to act（想 → 帮做）**：Thought 让模型拆解目标、追踪进度、处理异常、决定"下一步搜什么 / 干什么" → 治"死法 A 不会想"
> - **Act to reason（做 → 帮想）**：Action 把外部真实信息灌进 history、纠正/更新模型的内部想法 → 治"死法 B 不接触现实"
>
> ReAct 不是第三种新东西。它就是把两种朴素做法**让它们在每一步互相纠偏**而已。这种"朴素到像废话"的招能成立，本身就是个值得品的现象。

## 4. 实证：数字 — 以及更重要的"ReAct 自己怎么崩"

到这里你已经知道 ReAct 是什么、为什么管用。下一个该问的是：**它能管用到什么程度、它自己又会怎么崩**。

后半句尤其关键。很多二手摘要只会写 "ReAct 比 CoT 好"，但 ReAct 论文真正值得读的地方，是它**诚实地给出了自己的失败模式**。这些失败模式直接定义了后面 [[plan-and-solve]] / [[tree-of-thoughts]] / [[reflexion]] / [[react-failures]] 要解决的问题。

### 成功的数字（先交代一下）

- **知识密集型任务**（HotpotQA<Cite id="yang2018hotpotqa" />、Fever）：ReAct 接一个简单的 Wikipedia API（就 `search` / `lookup` / `finish` 三个动作）。Fever 上 60.9% vs CoT 56.3%
- **交互决策任务**（ALFWorld<Cite id="shridhar2020alfworld" />、WebShop<Cite id="yao2022webshop" />）：ReAct **绝对成功率 +34%（ALFWorld）/ +10%（WebShop）** 胜过用 10³~10⁵ 条数据训练的模仿学习 / 强化学习方法 — 而 ReAct **只用了 1~2 个 in-context example**

第二条尤其震撼：一个不需要任何专门训练、只在 prompt 里给两个例子的方法，**在交互任务上吊打专门训练出来的模型**。这是 in-context learning 时代关于"prompt 能走多远"的一个分水岭式证据。

### 失败模式：Table 2 的人工标注 — 这才是金子

作者人工标注了 200 条轨迹（ReAct 和 CoT 各 100 条，对错各半），把它们的成功 / 失败原因分类。结果对照非常有教育意义：

| 模式 | ReAct | CoT |
|---|---|---|
| ✅ True positive（推理和事实都对） | 94% | 86% |
| ⚠️ **False positive**（幻觉但蒙对了） | **6%** | **14%** |
| ❌ **Reasoning error**（推理错，含重复循环） | **47%** | **16%** |
| ❌ Search result error（搜不到有用信息） | 23% | — |
| ❌ **Hallucination**（事实幻觉） | **0%** | **56%** |
| ❌ Label ambiguity（答对了但标签没对上） | 29% | 28% |

> [!warning] 三条要刻进脑子的结论
> **1. ReAct 几乎不幻觉（0% vs CoT 56%）**。
> 每一步都被真实 observation 锚住了 — 这是 ReAct 最大的卖点。从此之后所有"减少幻觉"的方案都绕不开"让模型在生成时接触外部信号"这个思路。
>
> **2. 但 ReAct 的 reasoning error 反而更高（47% vs 16%）**。
> 论文的解释一句话讲完：
> > 交替"想-做-看"这个**结构约束**虽然让 ReAct 更可靠，但也**限制了它组织推理步骤的灵活性** — 它被迫一步一动，没法像 CoT 那样自由地长篇推演。
>
> 也就是说 ReAct 用"可靠"换"灵活"，而灵活性的损失会以 reasoning error 的形式付出代价。
>
> **3. 一个 ReAct 特有的失败：repetitive loop（重复循环）**。
> 模型反复生成上一次的 thought 和 action，跳不出循环。这是论文归类在 reasoning error 下的一个小项，但**它会单独成为后面 Reflexion 那一节的全部动机**。

### 这三条失败为什么重要

如果你只记住"ReAct 比 CoT 好"，那 ReAct 之后的所有研究你都会觉得是"为创新而创新"。但如果你记住**上面这三条失败**，后面的所有范式都会变得自然：

> [!example] 这三条失败串起整本讲义
> - **「结构约束 → reasoning error 高 / 没全局规划」** → 在动手前先列完整计划（[[plan-and-solve]]）、或把多条思路做成搜索树用价值函数剪枝（[[tree-of-thoughts]]）
> - **「跳不出重复循环」** → 失败后用自然语言反思、把教训写进记忆，下次别再犯（[[reflexion]]）
> - **「重复的 thought / action 塞满 context」** → 这正是 [[context-failure-modes]] 里 **Distraction** 和 **Poisoning** 在微观尺度的源头
> - **「Search result error 占 23%，工具拉胯 agent 就 derail」** → 工具的"对 agent 友好程度"成了瓶颈 → [[function-calling]] / [[memory-basics]]

换句话说，**ReAct 不是终点，是起点**。后面整本讲义都在补它身上这几个洞。

## 5. ReAct 和今天的 Function Calling 是什么关系

你现在用 Anthropic / OpenAI SDK 跑 tool calling 的时候，并不会手写 `Thought:` / `Action:` / `Observation:` 这种文本协议。但这不意味着 ReAct 过时了 — 恰恰相反，**它被以结构化的形式吸收进了 API 本身**：

| ReAct 原始形态（2022） | 今天的对应物 |
|---|---|
| `Thought: ...`（自由文本想法） | reasoning / thinking tokens（模型的"思考段"） |
| `Action: Search[...]`（文本里约定的动作格式） | **function calling / tool use**（结构化 JSON tool_call） |
| `Observation: ...`（把环境返回拼回 prompt） | tool result 消息回填到对话历史 |
| 手写 while-loop 解析 `Thought:` 这种 prefix | SDK 内置的 agent loop / tool-use 循环 |

```json title="一个现代 tool_call 长什么样"
{
  "tool_calls": [{
    "function": {
      "name": "web_search",
      "arguments": "{\"query\": \"Apple Remote original device\"}"
    }
  }]
}
```

换句话说，**你每次用 SDK 跑一个多步任务，底层在干的就是 ReAct 的事**。模型看着 history（thought + action + observation 的序列），决定下一步是再调工具、还是输出最终答案 — 一字未改。

ReAct 是单 Agent 范式的**地基**。后面 Plan-and-Solve、Tree of Thoughts、Reflexion，全都是在这个地基上加东西，不是替代它。看到任何一个新范式时，你都可以问一句：**"它在哪一步、用什么手段、改了 ReAct 的哪个洞？"** 你会发现答案永远是清晰的。

详见 [[function-calling]]。

## 走之前，再敲一遍

不管什么时候忘了 ReAct，回来抓这六条就行：

1. **要回答的问题**：agent 主循环里 `step = llm(history)` 那一行，模型怎么决定下一步？ReAct 是单 Agent 范式里**最朴素**的答案
2. **核心招**：把语言也当成一种动作 — $\hat{A} = A \cup L$，thought 是个"不改环境、只更新 context"的内部动作
3. **治哪两种朴素病**：纯行动"不会想"、纯推理"不接触现实" — 靠双向协同同时压住
4. **最大优点**：几乎不幻觉（0% vs CoT 56%），因为每一步都 grounded 在真实 observation 上
5. **它自己的病**：结构约束 → reasoning error 47% + repetitive loop + 检索质量是瓶颈 — 这三条直接定义了后面四节文章
6. **和今天的关系**：现代 function calling 多步循环 = 结构化版的 ReAct
