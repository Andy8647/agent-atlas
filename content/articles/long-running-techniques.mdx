# 长程任务三大技术

[[context-design]] 讲了 context 部件该怎么设计 — system prompt、tools、examples、just-in-time retrieval。但它隐含了一个前提 — **任务的总 token 量是能装进窗口的**。

当任务长到**总 token 数本身就超过窗口**时怎么办？例子：

- 一个大代码迁移：要读 50 个文件、改 30 个文件、跑测试、修错、最后写文档 — 一个文件几千 token，几小时下来轻松 200K+
- 一个深度研究任务：搜 100+ 篇文章、做笔记、综合、写报告 — 总信息量可能上 MB
- 一个长对话支持：用户跟 agent 聊几小时、几十轮对话 — 历史本身就撑爆窗口

这一篇讲 Anthropic 把三种长程技术**正式化**成工程纪律：**Compaction（压缩）/ Note-taking（记笔记）/ Sub-agent（子任务）**。

注意一件事 — 这三种技术**正是 [[memgpt]] 那一篇里 MemGPT 用"操作系统"机械实现过的三件事**。这里被抽象成面向**所有** agent 的通用做法，不绑死 MemGPT 的具体架构。

## 1. Compaction — 压缩历史，留干货

最基础的长程技术：**当 history 接近 context 上限时，把它摘要成精华，用摘要重启一个新窗口**。

```text title="Compaction 流程"
Round 1-30: history 累积到 180K token (接近 200K 上限)
            ↓
            触发 compaction:
            ↓
摘要器 LLM 读完 180K → 输出 12K token 的精华:
            - 当前任务进度
            - 关键决策点
            - 还没解决的问题
            - 重要的中间结果
            ↓
新 context = system_prompt + compaction_summary + 最近 3 轮
≈ 15K token (从 180K 压到 15K)
            ↓
Round 31 继续 — 模型看着摘要继续工作
```

### Compaction 的核心问题：留什么、丢什么

最关键的工程问题不是"怎么压"，而是**"压缩时保留哪些信息"**。Claude Code 给出的实践标准：

| 保留 | 丢掉 |
|---|---|
| **架构决策** — 为什么选 React 而不是 Vue | 工具调用的原始返回（已经被消化了） |
| **未解决的 bug** — 在 line 234 有一个待修的空指针 | 已经成功的中间步骤 |
| **实现细节** — 当前正在用的 API 接口 schema | 重复或相似的信息 |
| **用户的偏好和约束** — "不要引入新依赖" | 模型自己的中间 thought 链 |

> [!tip] 最安全、最轻的一种 compaction：tool result clearing
> Compaction 设计起来很难（"留什么 vs 丢什么"是个 judgment call）。但有一种特别低风险的轻量版本叫 **tool result clearing**：
>
> 一个工具结果在历史深处被用过了 — agent 已经基于它做了下一步决策，那个原始返回里的细节再也不会被引用了 — **何必再让 agent 看一遍原始返回**？
>
> 直接把那个 tool result 从 history 里删掉、只留 "tool X 已调用并被使用" 的 placeholder。这种 compaction 几乎不会丢信息（已经被消化的不需要再看），但能省大量 token。这是 [[aci]] 里"token 效率"那条原则的运行时延伸。

### Compaction 适合什么任务

- 需要**大量来回对话**的任务（深度调试、研究、迭代开发）
- 任务有**渐进感** — 越往后越接近答案，前面的过程可以浓缩
- 你能写一个**清晰的"留什么"标准**

不适合的情况会在 [[context-failure-modes]] 里讲 — 如果 context 已经被 **Poisoning（中毒）** 了，compaction 把毒一起浓缩进新摘要，越压越毒。所以 compaction 不是万能解药。

## 2. Note-taking — Agent 自己写笔记

第二种长程技术：**agent 定期把笔记写到窗口外的存储，之后再拉回来**。

这一招非常 underrated — 它把 [[memgpt]] 里 MemGPT 的 "Working Context"（常驻关键事实）从一个绑死在 MemGPT 架构里的特性，变成任何 agent 都能用的简单模式。

### 一个让人惊讶的例子

最有说服力的例子是 Anthropic 团队跑的 **Claude 玩 Pokémon** 实验：

- Claude 被让玩宝可梦红/蓝，从开局打到联盟冠军
- 任务**长达几千步**，远超 context window
- 提供了一个简单的工具：可以读 / 写一个 `NOTES.md` 文件

实际发生的事：

```text title="Claude 自己长出来的笔记习惯"
=== NOTES.md (Claude 自己写的) ===

## 当前目标
- 攻关常磐道馆 (草系道馆)
- 主力火属性宝可梦,需要小火龙练到 20 级以上

## 战绩
- Route 1: 探索完成 (1234 步)
- Route 2: 探索完成 (1567 步), 抓到了皮卡丘
- Pewter City Gym: 已通关
- Mt. Moon: 已通关,获得月之石
- Route 3-4: 探索完成
- Cerulean City: 当前位置

## 待办
- 找传送大师交换皮卡丘换雷电鸟道具
- Route 24 NUGGET 桥头 — 一连串训练家,可以一次性练级

## 重要事实
- 我的金钱: 8430 元
- 红色尝试过去乱入,失败了 (战力不足)
- Bill 的电脑在 1F,送了我"S.S. Anne 票"
```

这里有 3 个值得拆开看的细节：

1. **Claude 自己长出了这个习惯** — 它不是被显式教"你必须记笔记"，是发现"我的 context 装不下整局游戏" → 自己开始用工具写笔记
2. **笔记的结构 Claude 自己设计** — Markdown 标题分类、bullet list、状态标签 — 都是从游戏需求长出来的
3. **Compaction 之后能"复活"** — context 重置后，Claude 读自己的 NOTES.md 接着玩几小时

### Note-taking 适合什么任务

- 有**清晰里程碑**的迭代开发（"完成 step 1 / step 2 / step 3"）
- 需要在长程任务中**维持状态**的（游戏、研究、长项目）
- **结果可以被结构化** — 笔记不是流水账，能被组织成清晰的"我当前在哪、还要做什么"

### 一个实操细节

不需要给 agent 复杂的 note-taking 框架。一个 `read_file('NOTES.md')` + `write_file('NOTES.md', new_content)` 就够了。配上一句 system prompt：

```text title="一句话教会 agent 用笔记"
对于长程任务,你可以用 NOTES.md 记笔记。建议:
- 定期更新当前进度和待办
- 遇到关键决策点时记下来
- compaction 之前确保关键状态都在 NOTES.md 里
```

剩下的让模型自己长出来。

## 3. Sub-agent — 把任务分给"干净窗口"的子 agent

第三种技术是**架构层面**的 — 不再是给一个 agent 加 token 管理技巧，而是**把任务拆给多个 agent**。

### 核心机制

```text title="Sub-agent 模式"
主 agent (orchestrator):
  ├── 高层计划: "我要做 X, 分三步: A, B, C"
  ├── 派子 agent: spawn_subagent("做 A 这件事")
  │     ↓
  │   子 agent 1 用干净的窗口:
  │     - 自己读文件、调工具、做 A
  │     - 烧 80K context 完成 A
  │     - 回传 2K 的「干净摘要」给主 agent
  │     ↓
  ├── 主 agent 只看到 2K 摘要 (而不是子 agent 那 80K)
  ├── 派子 agent 2 做 B → 同样回传 2K
  ├── 派子 agent 3 做 C → 同样回传 2K
  └── 综合 3 个 2K 摘要 → 最终答案
```

### 为什么这个机制能省 context

两点：

- **每个子 agent 都用干净窗口** — 主 agent 不需要把"做 A 的所有中间过程"塞进自己的 context。子 agent 在它的小窗口里处理完，只把蒸馏后的结果回传
- **可以并行** — 3 个独立子任务能同时跑，时间可以缩短到原来的 1/3

实证支撑：Anthropic 在 *How we built our multi-agent research system*<Cite id="anthropic2024multiagent" /> 里报告，在复杂研究任务上多 agent 架构比单 agent **+90.2%**。

### 一个关键设计：回传摘要而不是原始 trace

第一次实现 sub-agent 时最常见的错误 — 把子 agent 的完整工作轨迹直接回传给主 agent。80K 子 trace 进主 agent 的 context，立刻爆掉。

正确做法是 **子 agent 自己生成浓缩摘要再回传** — 主 agent 看到的不是 trace，是 "完成了 A，关键结果是 X / Y / Z，遇到的坑是..." 这种 2K 的摘要。这正是 [[reflexion]] 里 Self-Reflection 把失败 trace 压成一句反思的同款思路。

> [!warning] Sub-agent 还有另一个视角 — 留到 cluster 5
> 这一节讲 sub-agent 只讲了它作为 **context engineering 工具** 的一面：用干净窗口避开 context rot。
>
> 但 sub-agent 还有完全独立的另一个视角 — 它是**多 agent 架构的基本单元**。同样是 "主 agent 管高层、子 agent 做深活、回传摘要"，从架构视角看叫 **Orchestrator-Workers**（[[workflow-vs-agent]] 里讲过的 5 种 workflow 之一）。
>
> 多 agent 视角带来一场对撞 — Anthropic 派认为多 agent 是复杂任务的解，Cognition 派认为它引入更糟的"上下文碎片化"。这场争论是 [[multi-agent-debate]] 的全部内容，这里不展开。

## 4. 三种技术的对应关系

三种长程技术正是 [[memgpt]] 里 MemGPT 三个机械动作的"正式版本"：

| 这一篇的技术 | MemGPT 里对应的机制 | 适合的任务 |
|---|---|---|
| **Compaction（压缩）** | 递归摘要（recursive summary） | 需要大量来回对话的任务 |
| **Structured note-taking** | 写进 Working Context / Archival | 有清晰里程碑的迭代开发 |
| **Sub-agent（子 agent 架构）** | （MemGPT 没有 — 这是新加的一层） | 需要并行探索的复杂研究 |

第三行特别值得注意 — **Sub-agent 是这套技术里 MemGPT 没有的一层**。MemGPT 假定一个 agent 自己管自己的记忆；sub-agent 把这个假设打破 — **多个 agent 协作**才能解决某些任务。

这也是这一篇通向 cluster 5（[[multi-agent-debate]] / [[harness-overview]]）的桥 — 当你接受"agent 不一定是单数"这个前提，整个系统设计的可能性空间就完全不一样了。
