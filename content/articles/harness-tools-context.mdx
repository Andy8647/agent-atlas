# 工具流水线与 Context Manager

[[harness-loop]] 把主循环外壳和请求构造拆完了。这一篇拆 harness 里两个**真正决定 agent 跑得稳不稳**的部件 — **工具执行流水线**（部件 ②）和 **Context Manager**（部件 ③）。

这两个部件是 cluster 3 / 4 全部内容在 harness 里的实现入口：

- [[function-calling]] / [[aci]] 关于**工具该怎么用**的所有原则 → 在工具流水线里被执行
- [[memory-basics]] / [[memgpt]] / [[context-engineering]] / [[long-running-techniques]] 关于**context 该怎么管**的所有原则 → 在 Context Manager 里被实现

## 1. 工具执行的 6 步流水线（Codex 风格）

Codex<Cite id="fulton2025codex" /> 把工具执行做成一个**显式的 6 步标准流程**，每一步都对应一种特定的失败模式。这条流水线是 [[function-calling]] / [[aci]] 那一堆原则在代码里的样子：

```text title="Codex 工具执行 6 步流水线"
LLM 输出 tool_call
       ↓
① 解析参数 (Parse)
   - JSON Schema 验证
   - 参数类型 / 必需字段检查
   - 失败 → 不执行,返回结构化错误给模型
       ↓
② 查权限 (Authorize)
   - 这个工具在当前 session 允许调吗?
   - 失败 → 触发审批流程或拒绝
       ↓
③ 沙箱执行 (Execute in sandbox)
   - 在隔离环境里跑工具
   - macOS Seatbelt / Linux Landlock
       ↓
④ 抓输出 (Capture)
   - stdout / stderr / exit_code / timing
   - 超时由 sandbox 强制
       ↓
⑤ 截断 (Truncate)
   - token 感知截断 — 不是按字节
   - 保留头 + 尾, 省略中间
       ↓
⑥ 格式化回填 (Format & inject)
   - 加上 tool_call_id 链回上一步
   - 写入 history
       ↓
继续主循环
```

逐个看几个最值得说的步骤：

### 步骤 ① 解析参数：JSON Schema 验证不只是"语法对"

很多人觉得这一步就是 JSON 语法检查。错。它还包括：

- **类型检查**：`limit` 是 number 还是 string？模型偶尔会写 `"limit": "20"`
- **必需字段**：模型有时会漏 `query` 这种必需参数
- **enum 验证**：`order_by` 必须是 `'asc'` 或 `'desc'`，模型可能写 `'descending'`

这些检查**失败时不该崩** — 应该把错误**写成对模型友好的形式**回填给 history：

```text title="参数错误回填示例"
工具调用失败:
  你调用了 search_logs,但缺少必需参数 'query'.
  schema: { query: string (required), limit: int (default 20) }
  
请重新调用并提供 query 参数.
```

这正是 [[aci]] 里"错误信息要可执行"原则的具体实现 — agent 看到这条错误**知道下一步怎么改**。

### 步骤 ③ 沙箱执行

`shell_command(rm -rf /)` 这种命令不能直接在用户机器上跑。生产 agent 必须有**沙箱隔离**：

| 平台 | 沙箱实现 |
|---|---|
| **macOS** | Seatbelt（`sandbox-exec`）— 通过 profile 文件限制文件系统、网络、IPC 访问 |
| **Linux** | Landlock + seccomp — 内核级文件系统访问控制 |
| **跨平台** | Docker / Firecracker — 完整虚拟化 |

沙箱不只是"防恶意"，也是"防 agent 自己写错"。一个意外的 `find / -delete` 没有沙箱兜底就完了。

### 步骤 ⑤ 截断：token 感知 + 保头保尾

工具返回可能很大 — 一个 `cat large_log.txt` 可能返回 50K 行。直接全塞 context 会爆。

Claude Code 的做法：**默认把工具返回截到 25k token**。超过部分按 **保头 + 保尾 + 省略中间** 的策略：

```text title="Token 感知截断 · 5000 行输出的处理"
原始输出 (5000 行, ~80K token)
       ↓ 截断为
[ 前 100 行 (~1.6K token) ]
...
[ 中间 4800 行 omitted ]
...
[ 后 100 行 (~1.6K token) ]

总计 ~3.2K token, 给 agent 留 21.8K 预算
```

为什么是头 + 尾保留、删中间？因为 [[context-failure-modes]] 里讲过的 **Lost in the Middle** — 模型对长 context 的中间部分召回最差。**既然中间最容易被忽略，截断就专挑中间删**，保住首尾。

这是 [[aci]] 里"token 效率"原则和 [[context-failure-modes]] 里 U 形召回曲线在代码里的实现。

### 步骤 ⑥ 格式化回填：tool_call_id 链工具调用

[[harness-loop]] §2 讲过 — history 用 `tool_call_id` 把工具调用和结果链起来。这一步执行的就是那条规则：

```python title="工具结果回填到 history"
history.append({
    "role": "tool",
    "tool_call_id": original_call.id,    # 链回 LLM 那一轮的 tool_call
    "content": truncated_output,
    "is_error": exit_code != 0,           # 标记是否是错误
})
```

注意 `is_error` 这一项 — 让模型一眼能看到 "工具失败了"，不用从 output 文本里推断。

## 2. Context Manager：harness 的无名英雄

Context Manager 是 harness 里**最不起眼但承担最多压力**的部件。它管的事：

- **token 计数**：随时知道当前 context 用了多少 token
- **自动 compaction**：超过阈值时自动触发摘要
- **context diffing**：只发变化的部分，提高 cache 命中
- **临时缓存清理**：把"已经被用过的"tool result 标记为可清

挨个看。

### Token 计数 — 简单但要准

最朴素的 token 计数：

```python title="粗略 token 计数"
def rough_token_count(text):
    # 经验值: ~4 字符/token (英文), ~2 字符/token (中文)
    return len(text) // 3   # 安全估计偏高一点
```

但生产里要用模型自己的 tokenizer（OpenAI 的 `tiktoken`、Anthropic 的 `anthropic.count_tokens`），因为 4-字符近似在长文本里能差出 10%+，对接近 context 上限的场景是致命的。

### 自动 compaction：Claude Code 的 `wU2`

Claude Code 有一个叫 `wU2` 的内部组件 — 当 context 使用到 **~92%** 时自动触发：

```text title="wU2 自动 compaction 流程"
监控:
  current_tokens / max_tokens > 0.92  →  触发

执行:
  1. 调一个专门的 summarizer LLM 读完整 history
  2. 生成摘要 — 保留:
     - 用户的原始任务
     - 关键决策和决策理由
     - 未解决的问题
     - 最近 3-5 轮对话
  3. 重要信息可能被挪进一个常驻 Markdown 文档 (类似 [[memgpt]] 的 Working Context)
  4. 替换 history 为 summary + 最近 N 轮

效果:
  context 从 ~92% 降到 ~30%, agent 继续工作, 用户察觉不到
```

注意几个细节：

- **阈值 92% 而不是 100%** — 留一些 buffer 给当前 turn 的 tool result 和模型回复
- **摘要任务用一个**独立的 summarizer LLM — 不是当前 agent 自己做（避免它把任务进度搞乱）
- **重要信息挪进 Markdown** — 这正是 [[memgpt]] 里 Working Context 的设计在 harness 里的实现

这是 [[long-running-techniques]] 里 Compaction 和 Note-taking 两条技术合起来的工程实体。

### Context Diffing：每轮只发变化的部分

[[prompt-caching-infra]] 那篇会详讲 cache 机制。这里先点出 Context Manager 的一个关键能力：

```text title="Context Diffing 示意"
上一轮发给模型的 prompt:
  [system_layer_A_B_C_D] [history_1...10]

这一轮的 prompt:
  [system_layer_A_B_C_D] [history_1...11]    ← layer A-D 一字未变, history 只多了 1 条

Context Manager 看到:
  - 前缀 [system_layer_A_B_C_D] [history_1...10] = 上一轮已经发过
  → 用 prompt_cache_key 告诉 API "这段你已经算过 KV cache 了, 复用就行"
  → 真正要新算的只有 history_11 这一段
```

这一节是 [[prompt-caching-infra]] 的预告 — Context Manager 在做的事就是给推理引擎喂"哪些是稳定前缀、哪些是新内容"的元信息。每节省一份重复计算，就是省一份钱和延迟。

## 3. TodoWrite：让 agent 维护一个常驻任务表

[[plan-and-solve]] 里讲过 "先规划再执行" 在 prompt 层是个轻量干预。harness 把它**常驻化** — Claude Code 内置一个叫 **TodoWrite** 的工具，让 agent 把当前任务列成结构化清单：

```text title="TodoWrite 维护的常驻 TODO"
[x] 读 auth.py 找 bug
[x] 找到 line 42 的空指针风险
[ ] 写测试复现 bug
[ ] 修复 bug
[ ] 跑测试确认修好
[ ] 提交 PR
```

关键设计：**每次工具调用后，harness 把当前 TODO 状态作为 system reminder 重新注入 prompt**。这样模型在每一轮都能看到"我当前在哪一步"，不容易跑偏。

这就是 [[plan-and-solve]] 那条 prompt 干预 + [[long-running-techniques]] 那条 note-taking 在生产 harness 里**合二为一**的形态：

- 既是 plan（规划 + 跟踪进度）
- 又是 note（写入 context 外的存储，但每轮回注）
- 模型自己决定什么时候 mark 一项为 done、什么时候加新任务

## 4. MCP：动态发现外部工具

最后一个值得点出的 Context Manager 相关部件 — **MCP（Model Context Protocol）**。

[[function-calling]] 那篇讲的工具是"启动时已经定好的"。MCP 是 Anthropic 推的开放协议，让 agent 在运行时**动态发现新工具**：

```text title="MCP 动态工具发现"
启动时:
  Claude Code 内置工具: read_file, write_file, bash, search_file...

用户连一个 MCP server (比如 GitHub MCP):
  Claude Code 自动发现新工具:
    - github.create_pr
    - github.list_issues  
    - github.review_comment
  把它们的 schema 加进当前 session 的 tools

agent 调 github.create_pr 时:
  Claude Code 通过 MCP 协议把调用转发给 GitHub MCP server
  收到结果回填进 context
```

这是 [[function-calling]] / [[aci]] 在生态层面的扩展 — 一个开放协议让**任何服务**都能成为 agent 的工具，不用 Claude Code 团队 hardcode 每个集成。

## 5. 一个完整的"一轮" — 把这些部件串起来

最后看一轮 LLM 调用从开始到结束在 harness 里走的完整路径：

```text title="生产 agent 一轮的完整路径"
1. 用户发消息 "查 auth.py 有没有 bug"
   ↓
2. Context Manager: 计算当前 context 大小
   - 92% 阈值未触发, 不需要 compaction
   ↓
3. 请求构造: 拼 system 指令 4 层 + history + TODO state
   ↓ 调 LLM (用 prompt_cache_key 复用前缀 KV cache)
4. LLM stream 返回:
   - text: "我先读一下这个文件"
   - tool_call: read_file("auth.py")
   ↓ 工具流水线 6 步
5. ① 解析 ②权限 ③沙箱执行 ④抓输出 ⑤截断 (保头保尾) ⑥回填 history
   ↓
6. 主循环判断: 模型还有更多 tool_call 要做 → 继续下一轮
   (回到步骤 2)
```

这一段是 [[react-core]] 那 6 行 `while llm(history)` 在生产 harness 里**真正发生的事**。
