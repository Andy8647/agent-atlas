# 记忆系统基础

[[react-core]] / [[reflexion]] / [[function-calling]] / [[aci]] 这几节里反复在撞同一堵墙：**agent 的一切都要塞进有限的 context window**。

- ReAct 的 history 越滚越长
- Reflexion 的反思记忆只能留 1-3 条（论文里设上限 $\Omega$）
- 工具返回得分页截断，不然撑爆 context
- ACI 的 token 效率原则整条都在和这堵墙搏斗

到这里要正面回答这个问题 — **当信息装不下时怎么办**？这是 [[agent-architecture]] 提到的三件套里第三条腿 **Memory** 的核心命题。这一篇先把"短期 vs 长期记忆"的基本框架立起来 — 它会成为 [[memgpt]] 和后面整个 cluster 4（[[context-engineering]]）的地基。

但**在讲解法之前**，先要破除一个直觉上的误解 — 它差点把整个 agent 记忆研究方向带偏。

## 1. 必须破除的误解：记忆 ≠ 把 context window 开大

直觉是这样的：

> 模型 context window 越来越大（GPT-4 从 8K 到 128K，Gemini 到 2M，Llama 4 Scout 号称 10M）。那"窗口够大不就不用记忆系统了？历史全塞进去就行了。"

**这个直觉是错的，而且错得很彻底**。两个原因：

### 原因 1：计算成本是窗口长度的平方

Transformer 的 self-attention 复杂度是 $O(n^2)$ — 窗口长度翻 10 倍，attention 计算量涨 100 倍。

实际表现就是 — 同一个模型用 4K context 和 128K context 跑同一个问题，后者的延迟和显存占用是前者的几十倍。

> [!note] 这是为什么 vLLM / SGLang 这类推理引擎存在
> KV cache 优化、PagedAttention、连续 batching、RadixAttention — 这些 infra 层的核心工作很大一部分就是在治这个 $O(n^2)$。
>
> 但**应用层（agent）不能等 infra 把成本压到零**。即使有一天 attention 真做到 $O(n)$，"不该塞的东西塞进去"的危害仍在 — 这是下面原因 2。

### 原因 2：长窗口模型用不好长 context（Lost in the Middle）

这一条比成本更要命。Liu et al. (TACL 2024)<Cite id="liu2024lostmiddle" /> 做了一个经典实验：

- 给模型一个长 context（比如 30 个文档拼起来）
- 把答案藏在某一个文档里
- **改变那个文档在 context 中的位置**，看准确率

结果不是均匀的，是一条 **U 形曲线**：

```text title="Lost in the Middle · 答案位置 vs 召回率"
召回率
 |  ●                                          ●
 |    ●                                      ●
 |      ●                                  ●
 |        ●     ●  ●  ●     ●  ●        ●
 |          ●●●          ●●●          ●●
 |
 +-----------------------------------------------> 答案在 context 中的位置
   开头                  中间                   结尾
```

**开头和结尾的信息记得住，中间的容易丢**。开窗口到 128K 不能改这条曲线 — 你的关键信息塞在 60K 位置就是会被忽略。

这条结论会在 [[context-failure-modes]] 里以 "Distraction（分心）"的名字再出现一次。它给整个 agent 记忆研究定了一个 mindset：

> **记忆的本质问题不是"窗口多大"，而是"有限的窗口里此刻该放哪几条"**。

这个 mindset 贯穿 [[memgpt]] / [[context-engineering]] / [[context-failure-modes]] 三节 — 后面所有的具体技术都是在回答"该放哪几条"的某一面。

## 2. 短期 vs 长期记忆：一个基本框架

Lilian Weng 2023 的经典博客<Cite id="weng2023agents" />借**人脑记忆**的分类给 LLM 记忆做了个映射 — 这是最常用的入门框架：

| 人脑记忆 | LLM 里的对应物 | 特点 |
|---|---|---|
| **短期 / 工作记忆**（STM） | **in-context** — 塞进 context window 的所有 token | 快、容量小、受窗口长度硬限制 |
| **长期记忆**（LTM） | **外部向量库 + 快速检索** | 容量近乎无限，但要"检索"才能用 |

具体到 agent 里：

- **短期记忆** = 你这次对话的 system prompt + 用户最近的提问 + 上几轮的 tool 调用 + tool 返回 + 模型的中间 thought
- **长期记忆** = 存在 PostgreSQL / 向量库 / 文件系统里的"agent 经验"：上次用户的偏好、历史对话摘要、知识库文档、agent 自己写过的反思

**关键差别**：

- 短期记忆**自动出现在 LLM 推理时的 context 里** — 它就是 context window
- 长期记忆**只在被"检索"时才进 context** — agent 需要主动调一个工具去外部库找

第二点是 agent 设计的核心 — **长期记忆是按需调入的，不是常驻**。一个 agent 可能"知道"用户的 100 万条历史对话（都存在向量库里），但任何一刻只把和当前问题相关的 5 条调进 context。这是 agent 用 8K 窗口处理"看似无限上下文"问题的基本机制。

## 3. RAG：长期记忆的标准管道

长期记忆怎么"快速检索"？最常见的实现是 **RAG（Retrieval-Augmented Generation）**<Cite id="lewis2020rag" />。

### Step 1 — Embedding：把文本变成向量

要做"语义检索"，光靠关键字 grep 不行 — 用户问"喝什么咖啡" 你的记忆里可能存的是 "hand-drip coffee preference"，关键字根本对不上。要让计算机能识别"语义相近"，得把文本变成另一种形式。

**Embedding 就是把一段文本压成一个固定长度的浮点数列表（向量）**：

```text title="Embedding 把语义变成几何"
"用户喜欢的咖啡是手冲埃塞俄比亚"
        ↓ embedding model (如 OpenAI text-embedding-3-small)
[0.023, -0.117, 0.842, ..., -0.054]    1536 维浮点向量
```

为什么是 1536 个数字？这是模型设计选择 — 模型内部学到的"语义维度"被压缩成这个长度。**核心性质**：

> 语义相近的文本，会被这个模型映射成**几何上接近的两个向量**。

具体地：

- "用户喜欢的咖啡是手冲埃塞俄比亚" 和 "hand-drip coffee preference" → 两个向量的距离**很小**
- "用户喜欢的咖啡是手冲埃塞俄比亚" 和 "今天上海天气怎样" → 两个向量的距离**很大**

注意 — embedding 模型本身是一个被预训练好的神经网络（比如 OpenAI / Cohere / BGE 等家提供）。**对你来说它是黑盒** — 你只需要调 API 把文本变成向量，不需要管它内部怎么学到这种映射的。

### Step 2 — 存储到向量库

把这些向量批量存到专门的向量数据库 — PostgreSQL 的 pgvector 扩展、或独立的 Chroma / Pinecone / Weaviate / Qdrant。它们的核心能力是**给一个 query 向量，快速找出库里距离最近的 top K 条**。

### Step 3 — 检索

给定 query 向量 $q$，要找库里"最近"的几条。最常用的距离不是欧氏距离而是**内积**（inner product）— 因为 embedding 模型训练时就是按"内积大 = 语义近"的约定优化的，用同款度量召回最准。这个搜索任务叫 **MIPS（Maximum Inner Product Search）**：

$$\text{top\_k} = \arg\max_{v \in \text{库}} \langle q, v \rangle$$

如果库有 100 万条向量，精确计算每对内积要 100 万次乘加。即使开 GPU 也撑不住实时检索。所以实际全用 **ANN（Approximate Nearest Neighbor）** 算法 — 牺牲 1-2% 精度换 100×–1000× 速度。

> [!note] ANN 算法（pgvector / Pinecone 都帮你包好了）
> 几个常听到的名字：**HNSW**<Cite id="malkov2018hnsw" />（多层小世界图，最常用）、**FAISS**（Meta 开源）、**ScaNN**（Google）。对应用开发者来说全是黑盒 — 你用 pgvector / Pinecone 时不需要选哪个，库默认会给你一个合理选择。**知道它们存在就行**。

实际写代码大概长这样：

```python title="一次 RAG 检索调用"
# 把 query 转向量
q = embed("用户上次喝什么咖啡？")

# 在向量库里找 top 5 最相似的记忆
results = memory_store.search(q, top_k=5)
# → [
#     "用户喜欢的咖啡是手冲埃塞俄比亚",
#     "上次推荐了 G1 耶加雪菲",
#     "用户对深烘焙不感兴趣",
#     ...
# ]

# 把结果塞进 prompt
prompt = f"基于以下历史记忆回答用户的问题:\n{results}\n\n用户问: ..."
```

模型推理时的 context = system prompt + **检索回来的 top K 记忆** + 当前问题。这就是为什么 agent 能"记住"500 次对话之前的事 — 那 500 次对话都存在向量库里，只在相关时被调进 context。

## 4. RAG 不能解决的问题（预告后面几节）

RAG 看起来很美 — 但它有几个硬限制：

| 限制 | 表现 |
|---|---|
| **检索器质量是天花板** | embedding 模型把"含义相近但词面不同"判断错 → 检索回来的不是真正相关的 |
| **被检索的内容不会自动相互组合** | 检索回 5 条独立片段，但**用户的真正问题需要把这 5 条综合**，模型不一定做得到 |
| **写入策略没指定** | RAG 只管"检索"，**到底什么该被存进长期记忆**？要 agent 主动决定 |
| **检索时机没指定** | agent 怎么知道"现在该去查长期记忆"？而不是凭已有 context 答 |

这 4 条里前 2 条是检索器底层的事；后 2 条是 agent 层面的工程问题 — **谁来决定写什么、什么时候读**。

[[memgpt]] 给的答案是：**让 LLM 自己管理自己的记忆**。Anthropic 在 [[context-engineering]] / [[long-running-techniques]] 里把这套机械实现抽象成 3 条通用纪律：**compaction（压缩）/ note-taking（记笔记）/ just-in-time retrieval（按需检索）**。

---

到这里 RAG 的"读"那条腿你已经清楚 — embedding + ANN 检索 + 把结果塞回 prompt。但 RAG 没告诉你**谁来决定写什么、什么时候触发读**。这是 agent 工程层面的问题，不是 retriever 层面的问题。

下一篇 [[memgpt]] 看一个非常具体的答案 — 它把整个 agent 的记忆管理类比成**操作系统的虚拟内存**，**让 LLM 自己用 function call 管理自己的记忆**。这个比喻不仅美，还能跑出 DMR 32% → 93% 的实证。
